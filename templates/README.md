## 📁 Directory Structure


**Key Principle**: Each source maintains its own namespace, preventing naming conflicts and making provenance clear.

## 🔧 How to Use

### 1. Using Slash Commands (AI Agents)

In your AI chat (Cursor, Windsurf, etc.), use slash commands to invoke templates:

```
/marketing_spec_kit.specify "User Authentication"
→ Uses templates/generic/templates/specify-template.md
→ Generates specs/001-user-auth/spec.md
```

### 2. Using CLI Commands

Some templates can be invoked via CLI:

```bash
marketing_spec_kit init my-spec.yaml
marketing_spec_kit validate my-spec.yaml
```

### 3. Manual Template Usage

You can also copy templates manually:

```bash
cp templates/generic/templates/specify-template.md specs/my-feature/spec.md
# Edit the copied file
```

### Option 2: Create New Source

Create a new source directory for domain-specific templates:

```bash
mkdir -p templates/custom/commands
mkdir -p templates/custom/templates

# Add your templates
templates/custom/templates/my-template.md
templates/custom/commands/my-command.md
```

## 🔍 Understanding the Structure

### Why Organize by Source?

1. **Clear Provenance**: Know which specification system each template comes from
2. **Namespace Isolation**: Different sources can have same-named commands without conflict
3. **Composability**: Mix and match specification systems as needed
4. **Maintainability**: Easy to update templates from upstream sources

### Example: Multiple Sources

```
templates/
├── generic/              # Universal patterns (0→1)
│   ├── commands/
│   │   └── specify.md
│   └── templates/
│       └── specify-template.md
└── spec-kit/             # Development patterns
    ├── commands/
    │   └── plan.md
    └── templates/
        └── plan-template.md
```

Both `generic` and `spec-kit` can coexist, each maintaining its own namespace.
